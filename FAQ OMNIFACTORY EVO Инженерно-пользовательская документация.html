🧠 Архитектурные слои системы

🖥️ UI Layer

Пользовательский визуальный конструктор. Отвечает за интерфейс выбора модулей, визуализацию pipeline, drag-and-drop узлы, отображение телеметрии и взаимодействие с API ядра.

builder_pro.html
⚙️ Core Layer

Главный вычислительный слой системы. Выполняет генерацию кода, обработку графов модулей, orchestration runtime, dry-run тестирование и сборку конечных ботов.

🌐 API Layer

FastAPI шлюз платформы. Обрабатывает запросы фронтенда, выдает ZIP-сборки, управляет runtime-ботами, telemetry API, registry endpoints и deployment pipeline.

🧩 Service Layer

Инкапсулирует вспомогательные сервисы: упаковку Docker, requirements generation, генерацию deployment package, менеджер модулей и экспорт проектов.

project_generator.py
🗄️ Data Layer

Слой моделей данных, telemetry state, Pydantic schemas, runtime metrics, Firestore/Neon интеграция и внутренние сущности системы.

models.py
🤖 AI Layer

AI/RAG подсистема. Подключает Gemini/OpenAI, ChromaDB, интеллектуальный поиск, knowledge-base routing и AI-консультантов.

🚀 Рекомендуемая инженерная стратегия

1️⃣ Разделить ядро

Вынести всю бизнес-логику из огромного server.py в отдельные сервисы и ядро compiler/runtime.

2️⃣ Создать Registry Engine

Все модули должны регистрироваться через registry.json, чтобы UI и backend были синхронизированы автоматически.

3️⃣ Ввести Runtime Graph

Построить node-based execution graph вместо линейной генерации кода.

4️⃣ Добавить Module SDK

Каждый модуль должен иметь: manifest, requirements, handlers, permissions и telemetry hooks.

5️⃣ Разделить UI

Перевести builder на полноценный SPA frontend: React/Vue/Svelte + websocket telemetry.

6️⃣ Подготовить Production Runtime

Docker swarm, isolated workers, async orchestration, container sandboxing и queue system.

🧬 Детальный инженерный разбор main.py / server.py

🚀 FastAPI Runtime Core

Главный серверный шлюз системы OmniFactory. Управляет всем жизненным циклом платформы: API, маршрутизацией, генерацией ботов, telemetry engine, background tasks и runtime orchestration.


В текущем состоянии server.py выполняет одновременно:

  • роль API gateway
  • роль orchestrator runtime
  • роль deployment manager
  • роль telemetry сервера
  • роль UI backend
  • роль AI integration layer
FastAPI + Uvicorn + Async Runtime
🧠 Runtime Orchestrator

Orchestrator — ядро исполнения runtime-графов. Оно принимает конфигурацию ботов, node pipeline, граф переходов и выполняет orchestration логики.


Основные задачи:

  • загрузка bot graph
  • управление runtime state
  • подключение TelegramConnector
  • динамический запуск узлов
  • поддержка AI flow execution
engine/orchestrator.py
📦 BotFactory Engine

BotFactory — это генератор конечного Python-кода бота. Он собирает runtime-модули в единый aiogram/main.py проект.


Внутри реализовано:

  • code block injection
  • module registry
  • handler generation
  • dynamic dependency system
  • AI module attachment
services/bot_factory.py
🧪 Dry-Run Simulator

simulator.py нужен для проверки логики без запуска реального Telegram-бота.


Это крайне важный инженерный слой, потому что позволяет:

  • тестировать node-граф
  • отлавливать ошибки до деплоя
  • проверять переходы
  • эмулировать runtime execution
  • строить visual debugger
core/simulator.py

🛠️ Что необходимо улучшить архитектурно

Проблема Текущее состояние Рекомендуемое решение
Монолитный server.py Слишком много ответственности в одном файле. Разделить на: routers/, runtime/, telemetry/, deployment/, ai/
Дублирование логики Повторяются routes и deployment handlers. Вынести в service layer и использовать dependency injection.
State хранится в памяти Runtime state теряется при перезапуске. Redis + distributed cache.
Нет очередей задач Все работает внутри FastAPI loop. Celery / RabbitMQ / Redis Queue.
Нет sandbox isolation Боты запускаются в общей среде. Docker containers + isolated workers.
Нет plugin SDK Модули жестко встроены в код. Создать manifest-based plugin system.
